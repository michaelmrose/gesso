#+title: Gesso
#+author: Michael Rose
#+options: toc:2


Gesso is a small Clojure web application library for building server-rendered, live-updating applications with plain data, Hiccup, HTMX, SSE, and Biff.

It is not a SPA framework. It is not trying to hide HTTP. The goal is to make rich web interfaces feel like ordinary server-rendered Clojure applications, while still supporting live updates, reusable UI components, theme generation, and tested application structure.

Gesso is currently early software. The API is expected to become more stable as real applications are built with it.

* What Gesso is for

Gesso is for applications where:

* HTML is primarily rendered on the server.
* Application state lives on the server.
* UI updates can be expressed as fragments.
* HTMX is enough for most interactions.
* SSE is enough for live invalidation and refresh.
* You want Clojure data and functions instead of a client-side app framework.
* You want reusable UI components without giving up normal HTML semantics.

The reference application style is demonstrated by Gessokit, especially the Human Help example app.

* Design goals

** Server-rendered first

Gesso assumes the server owns the truth. Pages, panels, forms, and cards are rendered as HTML fragments from Clojure data.

The browser should not need to reimplement the application model.

** HTMX-shaped interactions

Gesso leans into HTMX instead of fighting it. Forms submit normally through HTTP. Buttons and links can request fragments. Server responses replace small parts of the page.

** Live updates without a SPA

Gesso Live uses SSE to notify connected browsers that some part of the UI may be stale. The browser can then ask the server for the relevant fragment again.

The important event is not “replicate every mutation to every browser.” The important event is “the state you are looking at may have changed; refresh the affected fragment.”

** Reusable components

Gesso includes component helpers for common HTML/UI patterns. Components are intended to produce ordinary Hiccup and ordinary HTML, not opaque widgets.

** Theme generation

Gesso includes a theme build path that can scan component CSS roots and generate application theme CSS.

This lets a starter kit or app combine framework component styles with app-specific component styles.

** Client continuity

Live fragment replacement can otherwise destroy useful browser-local state. Gesso Live includes continuity helpers for restoring browser-local context across swaps, such as:

* scroll anchoring
* focused input/caret state
* selected or in-progress inputs
* native ~<details open>~ state

This lets the server stay responsible for real application state while the browser keeps transient interaction state.

* Non-goals

Gesso is not trying to be:

* a React replacement
* a client-side router
* a general-purpose component compiler
* a CSS framework
* a GraphQL client
* a distributed database
* a magic state synchronization system

It is intentionally boring where possible: Clojure functions, Ring handlers, Hiccup, HTTP, HTMX, and SSE.

- Core ideas

** Pages

A page is ordinary server-rendered HTML.

In practice this usually means:

* a Ring route receives a request
* the route builds a context map
* application code queries data
* a view function returns Hiccup
* the response is sent as HTML

** Fragments

A fragment is a smaller renderable part of the page.

Fragments are useful for HTMX because the browser can request just the part of the interface that changed.

For example, an app might have fragments for:

* a toolbar
* a request list
* a request card
* a notification area
* a sidebar
* a search result panel

** Invalidation

Live applications do not need to push full HTML every time data changes.

Instead, a mutation can invalidate a logical scope. Browsers subscribed to that scope receive an SSE event and ask the server for the relevant fragment.

This gives you a simple mental model:

#+begin_src text
data changes
-> affected live scope is invalidated
-> browser receives SSE notification
-> HTMX refreshes the fragment
-> server renders fresh HTML
#+end_src

** Continuity

Fragment replacement is powerful, but it can be rude. If the browser replaces a whole list, it might accidentally lose:

* scroll position
* the focused input
* the caret position
* open details/accordion state
* partially typed local input

Gesso Live continuity exists to keep that browser-local state stable.

Example:

#+begin_src clojure
(require '[gesso.live.continuity :as continuity])

(def request-list-continuity
(continuity/preserve
{:scroll {:selector "[data-request-card]"}
:focus true
:boxes [(continuity/details-open
{:selector "details[data-request-card][data-accordion-value]"
:key-attr "data-accordion-value"
:single? true})]}))
#+end_src

This says:

* preserve scroll context using request cards as anchors
* restore focus/caret where possible
* preserve which native ~<details>~ card is open
* identify cards by their ~data-accordion-value~
* restore at most one open card

The open card is intentionally not server state. It is local browser continuity.

* Gesso Live

Gesso Live is the live-fragment layer.

At a high level it coordinates:

* fragment URLs
* SSE streams
* invalidation scopes
* HTMX triggers
* client-side continuity
* refresh behavior after reconnects or browser visibility changes

A typical live fragment needs:

* a stable DOM target
* a URL that can render the fragment
* an SSE event that can trigger refresh
* enough included state for the server to render the same view
* optional continuity settings for browser-local context

- Components

Gesso components are Hiccup-producing helpers.

They should be boring and inspectable. A component should generally render understandable HTML with useful attributes and predictable classes.

The component layer is not intended to hide the web platform. It is intended to make application views smaller, more consistent, and easier to test.

* Themes

Gesso supports theme CSS generation from one or more component CSS roots.

A Gessokit application may combine component CSS from:

#+begin_src text
Gesso framework components
Gessokit shared components
Gessokit example app components
User application components
#+end_src

A typical theme build task resolves the installed Gesso dependency, adds its source/resources to the Babashka classpath, and calls the Gesso theme builder.

Example shape:

#+begin_src clojure
((requiring-resolve 'gesso.build.themes/build!)
{:input-dir "resources/themes"
:output-file "resources/public/gesso/app-themes.css"
:component-css-dirs ["path/to/gesso/components"
"src/myapp/components"
"src/myapp/site/components"]})
#+end_src

Applications should build themes before startup and ideally rebuild them during development when theme or component CSS files change.

* Relationship to Biff

Gesso is built in the same spirit as Biff applications: small pieces, server-rendered HTML, Clojure data, and plain web primitives.

Biff provides a productive base for Clojure web apps. Gesso adds application/UI conventions around:

* reusable components
* theme generation
* HTMX-oriented fragment rendering
* SSE-driven live invalidation
* client continuity across fragment swaps

- Relationship to HTMX

HTMX is the browser interaction layer Gesso expects.

Gesso does not replace HTMX. It helps Clojure applications produce consistent HTMX-enabled markup and live fragment behavior.

The intended model is still normal HTTP:

#+begin_src text
browser event
-> HTTP request
-> Clojure handler
-> Hiccup render
-> HTML response
-> HTMX swap
#+end_src

Gesso Live adds:

#+begin_src text
server invalidation
-> SSE event
-> HTMX fragment refresh
#+end_src

* Relationship to client-side JavaScript

Gesso uses JavaScript where it is useful, especially for:

* SSE connection handling
* HTMX integration
* continuity capture/restore
* browser-local interaction state

But application logic should generally remain on the server unless there is a clear reason to move it into the browser.

* Example: preserving open request cards

A common live UI problem is preserving which card the user has open while the surrounding list updates.

The wrong fix is to make the server remember every open card as application state.

The better fix is:

* server remembers real board/filter state
* browser remembers local open-card state
* Gesso Live restores the open ~<details>~ element after fragment replacement

Example continuity configuration:

#+begin_src clojure
(continuity/preserve
{:scroll {:selector "[data-humanhelp-request-card]"}
:focus true
:boxes [(continuity/details-open
{:selector "details[data-humanhelp-request-card][data-accordion-value]"
:key-attr "data-accordion-value"
:single? true})]})
#+end_src

This keeps the application model clean while still making live updates feel stable.
