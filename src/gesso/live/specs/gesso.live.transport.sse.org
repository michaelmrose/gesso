#+author: OpenAI
#+startup: overview

* Overview

=gesso.live.transport.sse= is the Server-Sent Events transport adapter for =gesso.live=.

It owns:
- SSE stream endpoint behavior
- SSE wire framing
- local subscriber registration for SSE connections
- cleanup on disconnect
- transport-specific keepalive behavior
- reconnect-friendly SSE concerns

It does *not* own:
- application/domain subscription semantics
- XTDB query or transaction behavior
- subscriber matching semantics
- browser-side HTMX attrs
- cross-node event distribution

This namespace is a transport adapter, not the semantic center of the live system.

* Purpose

The live system needs some way to deliver normalized changed-entity events to connected clients.

For SSE, that means:

1. accept an HTTP request that opens an event stream
2. register a node-local subscriber whose =:send!= writes SSE frames
3. keep that stream alive
4. unsubscribe when the stream dies
5. deliver normalized live events over SSE framing

This namespace handles those concerns and nothing broader.

* Core Design

SSE is the first transport, but it should not shape the whole system.

The transport layer should assume:
- subscriptions are already built elsewhere
- normalized live events already exist
- the local bus already knows how to route events to subscribers

This namespace only adapts:
- subscription input from the HTTP stream request
- event output to SSE frames

* Data Model

** Stream request input

An SSE connection request needs enough information to construct a subscriber.

At the generic transport layer, that should be one of:

- a prebuilt application-defined subscription descriptor
- or a transport-safe encoding of one

This namespace should not invent the application’s subscription model.

Conceptual request input examples:

#+begin_src clojure
{:subscription "...encoded..."}
#+end_src

or after app-layer parsing:

#+begin_src clojure
{:kind :request
 :id "req-123"}
#+end_src

The exact subscription shape is app-defined and opaque to this namespace.

** Local subscriber

This transport creates a local subscriber of the form:

#+begin_src clojure
{:subscriber/id sub-id
 :subscription subscription
 :send! send-fn
 :meta {:transport :sse
        :request-id ...
        :connected-at ...}}
#+end_src

Where:
- =:subscription= is opaque app-defined subscription data
- =:send!= writes normalized live events as SSE frames to the open stream

** Normalized live event

This namespace expects normalized live events shaped like:

#+begin_src clojure
{:event "live-update"
 :changed changed
 :consistency-token token
 :data data}
#+end_src

The meaning of =changed= is not interpreted here.

** SSE frame

An SSE frame is a text payload with the standard SSE shape.

Conceptually:

#+begin_src text
event: live-update
data: {"changed":...,"consistency-token":...,"data":...}

#+end_src

The exact encoding may use EDN or JSON depending on the final browser/client decision,
but framing belongs here.

* Functions

** =default-path=

A small constant or helper representing the default SSE endpoint path.

#+begin_src clojure
default-path => "/gesso/live/stream"
#+end_src

This may remain configuration rather than a hardcoded constant, but the concept is useful.

** =event-name=

Return the SSE event name for a normalized live event.

#+begin_src clojure
(event-name event) => string
#+end_src

Usually:
- use =(:event event)=
- default at a higher layer if missing

This helper exists to keep framing functions small.

** =event-payload=

Return the payload value that will be encoded into the SSE =data:= line.

#+begin_src clojure
(event-payload event) => payload
#+end_src

Recommended payload shape:

#+begin_src clojure
{:changed changed
 :consistency-token token
 :data data}
#+end_src

The event name stays in the SSE =event:= field.

** =encode-payload=

Encode an event payload for the SSE =data:= field.

#+begin_src clojure
(encode-payload payload) => string
#+end_src

This namespace should own the transport encoding choice for SSE.

Initial implementation may use:
- =pr-str= for simplicity in prototypes

A later production choice may prefer:
- JSON for easier browser interop

The rest of the live system should not care.

** =event->sse=

Convert one normalized live event into an SSE frame string.

#+begin_src clojure
(event->sse event) => string
#+end_src

Expected conceptual output:

#+begin_src text
event: live-update
data: <encoded-payload>

#+end_src

This is the primary framing helper.

** =keepalive-frame=

Return a keepalive/comment frame.

#+begin_src clojure
(keepalive-frame) => string
#+end_src

Typical SSE keepalive shape:

#+begin_src text
: keepalive

#+end_src

This is useful to keep proxies and clients from treating the stream as dead.

** =subscriber-send-fn=

Build the =:send!= function for one SSE stream.

#+begin_src clojure
(subscriber-send-fn stream-state) => (fn [event] ...)
#+end_src

Responsibilities:
- frame normalized events with =event->sse=
- write them to the underlying stream
- remain transport-local
- not know matching semantics

** =build-subscriber=

Construct the local subscriber for one SSE connection.

#+begin_src clojure
(build-subscriber {:subscriber/id sub-id
                   :subscription subscription
                   :stream-state stream-state
                   :meta meta})
=> subscriber
#+end_src

Responsibilities:
- keep the subscriber shape consistent
- attach SSE transport metadata
- use =subscriber-send-fn=

** =request->subscription=

Turn the incoming SSE HTTP request into an application-defined subscription.

#+begin_src clojure
(request->subscription ctx) => subscription
#+end_src

This is the trickiest boundary.

There are two valid designs:

*** Option A: generic transport only
This function is supplied by app code/config.
Then this namespace stays fully transport-only.

*** Option B: thin default transport parsing
This namespace may parse a configured parameter like =subscription= and hand it back as opaque data.

The preferred reusable design is Option A:
Gesso should not own app subscription parsing.

** =open-stream-response=

Create the streaming HTTP response.

#+begin_src clojure
(open-stream-response ctx on-open on-close) => ring-response
#+end_src

This function should encapsulate transport-specific streaming mechanics.

Responsibilities:
- open SSE response
- run subscriber registration on open
- run cleanup on close
- schedule or support keepalives as needed

Exact implementation depends on the Ring/server stack.

** =handler=

Main SSE endpoint handler.

#+begin_src clojure
(handler ctx) => ring-response
(handler {:keys [ctx subscription-fn subscriber-id-fn ...]}) => ring-response
#+end_src

Conceptual behavior:
1. derive subscription from request using app-supplied hook or configured parser
2. allocate subscriber id
3. open stream response
4. register local subscriber with =gesso.live.bus/subscribe!=
5. deliver keepalives as needed
6. unregister on disconnect

This is the primary SSE entrypoint.

** =stream-url=

Build an SSE stream URL from an already-encoded subscription token or query shape.

#+begin_src clojure
(stream-url encoded-subscription) => string
#+end_src

This helper should remain small and transport-specific.

It should not invent semantic subscription shapes.

* Suggested Configuration Hooks

A reusable SSE transport will likely need app-provided hooks or config for:

- =:subscription-fn=
  - request -> opaque subscription

- =:subscriber-id-fn=
  - request -> new subscriber id

- =:path=
  - endpoint path

- =:keepalive-ms=
  - keepalive interval

- =:open-stream-fn=
  - server-specific streaming primitive

- =:encode-payload-fn= (optional)
  - payload -> string

This keeps domain and server-specific assumptions out of the transport core.

* Relationship to =gesso.live.bus=

This namespace should:
- create subscribers
- subscribe them to the local bus
- unsubscribe them on disconnect

It should not:
- maintain matching indexes itself
- decide which subscribers should receive which events

That is the bus + matcher’s job.

* Relationship to =gesso.live.match=

This namespace should not call matcher logic directly.

It hands opaque subscriptions to the bus.
The bus and matcher handle routing.

This separation is important.

* Relationship to =gesso.live.htmx=

This namespace should not generate HTMX attrs.

That belongs in =gesso.live.htmx= or higher-level UI helpers.

The only shared contract here is:
- HTMX/browser opens this SSE endpoint
- this namespace streams normalized live events

* Relationship to Cross-Node Distribution

This namespace is node-local.

If a write on node A results in a changed-entity event and that event is shared across nodes,
node B’s local bus may notify SSE subscribers connected to node B.

This namespace only handles the final hop:
- bus event -> SSE frame -> socket/stream

It is not the shared broker.

* Reconnect and Failure Semantics

SSE is inherently reconnect-oriented.

This namespace should assume:
- streams may die
- node-local subscriber state may disappear on node failure
- reconnect to another node is normal

Therefore:
- unsubscribe on disconnect should be safe and boring
- stream lifecycle should not be treated as durable truth
- clients should be expected to refresh from source of truth after reconnect if needed

This is one reason SSE is a good fit for invalidation-style live systems.

* Recommended First Implementation

The first real implementation should support:

- one SSE endpoint
- one opaque app-defined subscription per connection
- local subscribe on open
- local unsubscribe on close
- normalized event -> SSE framing
- optional keepalive comment frames

That is enough to validate the transport layer without overbuilding it.

* Non-Goals

This namespace should not:
- define what subscriptions mean
- parse arbitrary domain graphs
- decide who matches a changed entity
- own consistency semantics beyond forwarding event payloads
- become a generic browser client framework

* Design Notes

- SSE is the first transport, not the semantic center.
- This namespace should be replaceable or complementable by a WebSocket sibling later.
- The more opaque the app subscription remains here, the more reusable Gesso stays.
- The framing and stream lifecycle are the real responsibilities of this namespace.

* Summary

=gesso.live.transport.sse= is the SSE adapter for the live system.

It:
- accepts SSE stream requests
- turns them into local subscribers
- registers them with the node-local bus
- frames normalized live events as SSE
- cleans up on disconnect

It does not:
- define domain meaning
- define subscription semantics
- decide match behavior

That is exactly why it belongs in Gesso.
